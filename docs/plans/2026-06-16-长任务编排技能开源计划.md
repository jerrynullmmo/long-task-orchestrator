# 长任务编排技能开源计划

## 目标

把现有内部复杂工作流里的长任务机制抽取成一个通用代理技能，并整理成可开源仓库。首版面向 Codex 优先，兼容 Claude Code；内部工作流只作为机制来源，不把内部业务规则、路径、项目名或窗口号带入开源内容。

## 实施范围

- 新建 `long-task-orchestrator` 技能目录。
- 补齐 `SKILL.md`、参考文档、机器模板、校验脚本、脱敏示例和评测提示。
- 根目录补 `README.md`、`LICENSE` 和基础忽略规则。
- 不创建 GitHub 远端，不发布，不打标签。

## 设计原则

- 主线程只做编排、状态登记、任务包下发、回执读取和最终放行。
- 独立执行单元负责具体执行，可以是 Codex 新线程、子代理、命令任务或人工复制任务包。
- 执行单元必须拿到任务包，任务包必须写清允许写入路径、授权产物路径、禁止触碰路径和回执路径。
- 完成状态以机器可读回执和监控状态为准，不能用自然语言“完成了”直接放行。
- 高风险产物默认必须走独立线程；子代理只能做辅助判断、证据采集和非高风险复核。

## 验收标准

- 技能目录符合代理技能目录约定。
- 脱敏示例能通过最小校验脚本。
- 模板能被复制到生产项目使用。
- `SKILL.md` 能指导主线程完成分派判定、任务包创建、回执读取和闸门放行。
- 所有示例不包含组织路径、生产窗口号、业务项目名、内部主题或外部对象信息。

## 调研结论

现有开源项目可以借鉴，但不直接替代本技能：

- `long-task-harness`：借鉴项目状态目录和进度读取脚本。
- `context-packet`：借鉴文件化上下文包、依赖图和预算意识。
- `Continuous-Claude`：借鉴单任务执行代理和交接文件。
- `superpowers`：借鉴子代理隔离和复核纪律。
- `agent-teams`：借鉴任务拆分、依赖图和文件所有权。
- `OpenCLI`：借鉴技能加命令包装方式。

首版选择自建，因为上述项目没有完整覆盖主线程放行、任务包授权、执行单元隔离、机器回执、监控状态和闸门校验的组合。
