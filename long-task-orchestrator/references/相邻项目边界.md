# 相邻项目边界

## 直接结论

首版选择自建 `long-task-orchestrator`。原因是相邻项目各有价值，但没有一个完整覆盖本技能的交付协议。

## 可借鉴对象

| 项目 | 可借鉴点 | 不直接替代原因 |
|---|---|---|
| `long-task-harness` | 项目状态目录、进度读取脚本 | 偏会话连续性，不负责任务包授权和主线程放行 |
| `context-packet` | 文件化上下文包、依赖图、预算意识 | 偏上下文解析，不负责执行单元授权和回执闸门 |
| `Continuous-Claude` | 单任务执行代理、交接文件 | 偏开发计划执行，不覆盖多载体监控状态 |
| `superpowers` | 子代理隔离、复核纪律 | 偏软件开发方法，不是通用长任务交付协议 |
| `agent-teams` | 任务拆分、依赖图、文件所有权 | 偏团队编排策略，不提供任务包回执模板 |
| `OpenCLI` | 技能加命令包装 | 偏工具操作，不是长任务主控协议 |

## 本技能边界

本技能不是任务平台，不提供队列、后台服务、远程执行、账号权限或工作看板。它只提供代理可读、文件可落地、脚本可校验的长任务编排协议。
