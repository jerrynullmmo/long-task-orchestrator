# 长任务编排技能

`long-task-orchestrator` 是一个面向 Codex 优先、兼容 Claude Code 的代理技能，用来把容易漂移的长任务拆成可控的主线程编排流程。

它解决的不是“如何让代理一直跑”，而是“主线程如何可靠地分派任务、限制执行范围、读取机器回执，并用监控状态和闸门证据决定是否放行”。

## 使用效果

| 问题 | 使用后 |
|---|---|
| 主线程一边读材料、一边改产物、一边判断放行，容易上下文拥挤 | 主线程只保留目标、状态、授权和闸门判断 |
| 子代理或新窗口拿到太多上下文，容易擅自扩大范围 | 每个执行单元只拿任务包、必读文件和授权路径 |
| 执行方说“已经完成”，但主线程缺少可复核证据 | 完成状态必须落成机器回执和监控状态 |
| 高风险产物被轻量子代理直接改写 | 高风险产物默认必须走独立线程或等价隔离载体 |
| 多轮返工后不知道当前阶段是否能推进 | 闸门脚本先检查结构证据，再由主线程决定放行 |

## 适用场景

- 一个任务预计超过 15 分钟，且需要多个执行单元分工。
- 主线程上下文容易被长材料、排障、渲染、构建或审计过程塞满。
- 某些产物风险高，需要授权路径、禁止触碰路径和独立回执。
- 需要把子代理、新线程、命令任务或人工执行统一成“独立执行单元”。
- 希望执行方只回传结构化证据，而不是让主线程持续读取完整过程。

## 不适用场景

- 一次性小修、小文件编辑或单命令执行。
- 需要完整任务平台、队列系统、远程调度器或权限中心。
- 只想记录会话进度，而不需要任务包、回执、监控状态和闸门放行。

## 最小使用流程

### 第一步：主线程判断是否分派

主线程先判断任务是否值得拆出去。通常满足任一条件就分派：

- 任务时间长，主线程继续做会挤压上下文。
- 需要独立窗口、子代理、命令任务或人工确认。
- 产物风险高，需要独立授权和回执。
- 同阶段需要复核、挑刺或证据采集。

如果不分派，也要在调度记录里写清原因，例如“只读解释”“极小修改”“已有执行单元正在等待回执”。

### 第二步：建立状态包目录

在目标项目里建立一组状态文件。可以直接参考 `examples/最小闭环示例/状态包/`：

```text
状态包/
  当前状态.json
  调度记录.json
  任务包/
  回执/
  监控/
```

这些文件让主线程不用靠聊天记忆管理阶段状态。

### 第三步：复制任务包模板

复制 `templates/任务包.template.json` 到目标项目的 `状态包/任务包/`，然后填写最小字段：

| 字段 | 用途 |
|---|---|
| `任务包ID` | 绑定任务包、回执和监控状态 |
| `执行单元类型` | 标明是 `subagent`、`独立线程`、`命令任务` 或 `人工执行` |
| `上游状态版本` | 防止执行方拿旧状态交付 |
| `必读文件` | 限定执行方必须读取的材料 |
| `允许写入路径` | 限定执行方可以写哪里 |
| `授权产物路径` | 限定本次应产出的文件 |
| `禁止触碰路径` | 明确不可读写或不可改动的范围 |
| `回执监控路径` | 规定执行完成后写回哪里 |
| `监控器状态路径` | 规定主线程或自动化读取哪里 |
| `验收标准` | 写清主线程如何判断是否进入闸门 |

高风险产物只能授权给 `独立线程` 或用户确认的等价隔离载体。

### 第四步：把任务包下发给执行单元

下发时只给必要内容：

1. 任务包完整内容。
2. 必读文件路径。
3. 允许写入路径和禁止触碰路径。
4. 回执文件路径。
5. 完成后停止，不自行进入下一阶段。

这一步的关键是“少给但给准”。不要把主线程全部上下文交给执行单元。

### 第五步：执行单元写回执

执行单元完成后复制 `templates/回执.template.json`，写入任务包要求的回执路径。

主线程只接受机器可读回执，不接受自然语言完成声明。回执至少要能回答：

- 用的是哪个 `任务包ID`。
- 输入版本是否等于任务包里的 `上游状态版本`。
- 读了哪些文件。
- 写了哪些产物。
- 有没有失败项、阻塞问题和存疑事项。
- 闸门结果是什么。

### 第六步：主线程读取监控状态

监控状态可以由自动化写入，也可以由主线程检查回执文件后写入。最小字段参考 `templates/监控状态.template.json`。

只有 `监控状态` 等于 `已收到回执`，且 `任务包ID`、`回执路径`、`状态路径` 都匹配时，主线程才继续放行判断。

### 第七步：运行闸门脚本

在仓库根目录运行：

```bash
python3 long-task-orchestrator/scripts/validate_orchestrator.py \
  --project long-task-orchestrator/examples/最小闭环示例
```

预期输出包含：

```text
结果: 通过
```

脚本通过只代表结构证据合格，不代表业务自动通过。主线程仍要按验收标准做最终判断。

## 完整示例

最小闭环示例位于 `long-task-orchestrator/examples/最小闭环示例/`。

它展示了一个文档整理任务如何闭环：

1. 主线程写入 `当前状态.json` 和 `调度记录.json`。
2. 主线程创建 `任务包/文档整理任务.json`。
3. 独立执行单元写入 `回执/文档整理任务回执.json`。
4. 监控状态写入 `监控/文档整理任务监控.json`。
5. 主线程运行校验脚本后决定是否放行。

可以直接运行：

```bash
cd long-task-orchestrator/examples/最小闭环示例
python3 ../../scripts/validate_orchestrator.py --project .
```

## 脱敏自查

发布前可用本地私有清单做脱敏自查。先创建不入库的 `.redaction-keywords`，每行写一个敏感片段；需要正则时给该行加 `re:` 前缀。然后运行：

```bash
python3 long-task-orchestrator/scripts/check_redaction.py --keywords .redaction-keywords .
```

`.redaction-keywords` 已加入 `.gitignore`，不要把私有关键词或私有正则提交到公开仓库。

## 与相邻项目的边界

- `long-task-harness` 类项目偏会话连续性和进度日志。
- `context-packet` 类项目偏上下文包和依赖图解析。
- `superpowers` 偏软件开发方法和子代理执行纪律。
- `agent-teams` 偏多代理任务拆分和团队协作模式。
- 本技能只聚焦长任务交付协议：任务包、隔离执行、机器回执、监控状态和主线程放行。

## 常见问题

### 校验通过是否等于任务完成

不是。校验脚本只检查任务包、回执、监控状态和授权路径是否自洽。业务质量仍由主线程根据验收标准判断。

### 是否必须创建新线程

不一定。只读复核、证据采集和非高风险分析可以用 `subagent`。正式产物、高风险产物、发布前验收和重大返工才默认使用独立线程。

### 是否能当任务平台用

不能。这个技能不提供队列、远程执行、账号权限、看板或后台服务。它提供的是代理可读、文件可落地、脚本可校验的长任务交付协议。

### 为什么不用自然语言回执

自然语言完成声明很难稳定复核，也容易遗漏输入版本、授权路径和失败项。机器回执能让主线程用脚本先做结构校验，再做业务判断。
